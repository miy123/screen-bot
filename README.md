# Screen Bot

GUI 上方有個模式下拉選單，兩種模式共用同一組啟動/停止（F1/F2）：

- **影像辨識**（預設）：用螢幕截圖 + 圖像比對辨識遊戲畫面中的素材（樹、礦石…），自動移動過去、
  點擊按鈕採集，採不到時會重對準再試，找不到素材時回中心待機。
- **固定路線**：完全不做圖像辨識，照你手寫的固定腳本「移動幾秒 → 採集 → 移動幾秒 → 採集…」跑，
  適合素材密集、懶得處理辨識的區域。詳見下面「固定路線模式」。

兩種模式全部靠模擬滑鼠操作，不讀取遊戲記憶體。

> 這款遊戲目前吃不到模擬鍵盤輸入，所以移動／轉向／採集都改成用滑鼠點擊/按住畫面上固定的按鈕座標，
> 不是按鍵盤。詳見下面「控制方式」。

## 需求與安裝

```
pip install -r requirements.txt
```

需要 Python 3 + `pyautogui`、`opencv-python`、`pillow`、`keyboard`（打包用 `pyinstaller`）。

Windows 上 `keyboard` 套件需要系統管理員權限才能註冊全域熱鍵，不然 `F1`/`F2`/`F3` 熱鍵不會有反應。
這些熱鍵讀的是你自己實體鍵盤的按鍵（用來控制腳本本身），跟遊戲吃不吃模擬鍵盤是兩回事。

`bot.py` 開頭會自動檢查目前是不是系統管理員身分，不是的話會自己跳 UAC 視窗要求提權、重新啟動一份
有權限的自己（原本那個沒權限的行程會直接結束）——不用自己記得每次都要「以系統管理員身分執行」，
跳出 UAC 視窗時按「是」即可；打包後的 exe 額外內嵌了要求提權的 manifest（`build.bat` 的
`--uac-admin`），啟動時 Windows 會直接跳 UAC，效果一樣但少一次重新啟動的過程。

## 快速開始

1. 用 `capture_helper.py` 截圖，工具會先問你現在要截哪個素材／面向，自動存成對應檔名。
2. 用 `test_recognition.py` 拿截圖 + 模板測一下辨識效果，調到 `MATCH_THRESHOLD` 抓得穩。
3. 找出遊戲畫面上移動/採集按鈕的螢幕座標：開 `python bot.py`，按 `F3`（或按 GUI 上的「Debug 座標」鈕），
   在倒數的幾秒內把滑鼠移到目標按鈕上，log 會印出當下的座標，填進 `config.py` 的 `MOVE_POINTS`／`COLLECT_POINTS`。
   之後座標要重抓（換解析度、遊戲視窗搬過位置）也是用這個功能，不用改程式碼。
4. 打開 `config.py`，把素材圖片路徑、對應的 `collect_action` 等填好（詳見下面「config.py 設定」）。
   如果想用固定路線模式，改編輯 `FIXED_ROUTE` 即可，不需要素材圖片。
5. 執行 `python bot.py` 開啟控制面板，在「模式」下拉選單選「影像辨識」或「固定路線」。
6. `F1` 啟動、`F2` 停止（兩種模式共用）；滑鼠移到螢幕左上角可以緊急停止（pyautogui 的 failsafe）。
7. 固定路線模式下，`F5`/`F6`/`F7`/`F8` 可以分別單獨測試上/右/左/下其中一塊，不用每次都跑完整輪
   （四塊）才能看到那一塊的結果，`F2` 一樣可以停止；不管「模式」下拉選單目前選哪個，這四個熱鍵
   永遠是固定路線模式。

## 控制方式：移動用鍵盤或滑鼠、採集用滑鼠點擊

這款遊戲不接受「普通的」模擬鍵盤（`pyautogui.keyDown`/`press`、`keyboard` 套件的 `.press()`——這些都是
用 Windows 的 `SendInput` 送虛擬鍵碼，會被這款遊戲或其模擬器層過濾掉），真的實體鍵盤按鍵倒是完全正常
（`record_route.py` 就是靠這點運作的）。移動/轉向因此原本改成用 `pyautogui` 的滑鼠功能
（`mouseDown`/`mouseUp`）去按住畫面上固定的方向鈕座標，而不是按鍵盤。

**`config.USE_KEYBOARD_MOVEMENT`（預設 `True`）**：測試發現 `pydirectinput` 這個套件送出的也是
`SendInput`，但用的是硬體「掃描碼」（scan code）而不是虛擬鍵碼，很接近「看起來像真的鍵盤」，值得試試
看這款遊戲會不會接受——如果會接受，移動就能直接用 `config.MOVE_KEYS`（預設 WASD）按鍵盤模擬，不用再
靠滑鼠拖曳畫面上的方向鈕，理論上也不會再有滑鼠模擬移動速度跟真人鍵盤不一致的問題（`FIXED_ROUTE_RECORDED_SCALE`
已重設回 `1.0`，因為那個係數是針對舊的滑鼠重播方式調的，換成鍵盤重播後不適用，需要重新測）。如果實測
角色沒反應/亂轉，代表這款遊戲連掃描碼也擋，把這個設定改回 `False` 就會退回原本滑鼠拖曳方向鈕的做法
（`config.MOVE_ORIGIN`/`MOVE_POINTS`，見下面）。

採集動作(`config.COLLECT_POINTS`)不受這個設定影響，一律都是滑鼠點擊具名按鈕（例如 `"lumber"` 是伐木鈕、
`"ore"` 是採礦鈕），`MATERIALS` 每一項用 `collect_action` 指定要點哪一個；`config.JUMP_POINT` 是選用的
跳躍鈕座標，用在卡住脫困時，沒有這個按鈕就留 `None`，脫困時會跳過這一步。

**斜向移動**：`_hold_point()`（`engine.py`）的 `direction` 參數現在可以是單一方向字串，也可以是方向
清單（例如 `["up", "right"]`）——`USE_KEYBOARD_MOVEMENT=True` 時清單裡每個鍵會「同時」按住，就是真人
玩家同時按住兩個 WASD 鍵做斜向移動的效果；`record_route.py` 錄製時如果偵測到兩個方向鍵有重疊（其中一
個還沒放開、另一個就按下去了），也會正確錄成一個「同時按住」的步驟，而不是拆成兩個各自完整時長、
先後執行的動作（那樣會少錄到斜向、實際花的時間也會變兩倍）。滑鼠模式（`USE_KEYBOARD_MOVEMENT=False`）
沒有這個能力，只有一個游標同一時間只能按住一個按鈕，遇到清單會退回「每個方向各自完整按住一輪」的
近似做法（跟 `config.py` 算法產生 TOP 那塊斜向格線的做法一樣），花的時間是清單長度倍數，不是真正同時。

**影像辨識模式（`BotEngine`）逼近目標時仍然只能單方向**：`_move_toward()` 裡追蹤素材/地標的移動邏輯
沒有改，同一時間還是只挑「距離較遠的那個軸」單獨按住（例如目標在右上方，會先往右修正，等左右對齊得
差不多了才切換成往上修正）去逼近斜向目標，不是真正同時的斜向移動——固定路線模式已經不用這套。

這些座標是綁定「目前螢幕解析度＋遊戲視窗位置/大小」的，換解析度、視窗搬過位置、或視窗大小改變
都要重新抓一次座標，不然點擊的位置就不對了。

## 運作方式（影像辨識模式）

`engine.py` 的 `BotEngine` 是一個簡單的狀態機，主迴圈大致是：

```
尋找素材中 → 移動靠近中 → 採集中 → 尋找素材中 → ...
```

- **尋找素材中**：對 `config.MATERIALS` 每一種素材做樣板比對，找螢幕上距離角色（畫面中心）最近的一個。
  找不到的話，若 `RETURN_HOME_WHEN_IDLE = True` 就走回啟動時的位置待機，每隔
  `IDLE_SCAN_INTERVAL` 秒才重新掃描一次；設 `False` 則用舊的 `SEARCH_PATTERN` 到處遊走找。
- **移動靠近中**：鎖定這一輪選中的那個素材（同一個實體，不是同種類的任何一顆），持續按住方向按鈕靠近，
  每個 tick 直接依座標（dx/dy）決定方向、按壓固定的短脈衝時間，不套用下面的重對準策略；
  含卡住偵測（久了沒移動就跳一下、換方向逃脫）。
- **採集中**：點擊 `collect_action` 對應的採集按鈕，等一下再重掃一次確認素材是否消失。
  沒消失就「重對準」（面向素材的方向按鈕沒按對就補按——面向判斷存在的目的就是確保有對準素材才點得到；
  已經面向正確就從 `REALIGN_STRATEGIES` 挑一個策略執行——每個策略是一串「往哪個方向按住幾秒」的動作，
  見下面「重對準策略」）再靠近一次重試，最多重試 `COLLECT_RETRY_MAX` 次。這些策略只在這個「卡住/重試」
  情境下用到，平常移動靠近不會觸發。
- **同一位置持續失敗**：跨越好幾輪都採不到同一顆（位置在 `SAME_MATERIAL_TOLERANCE` 內視為同一顆），
  累積到 `STUCK_FAIL_THRESHOLD` 次會改用截圖判定面向（如果有設定，見下面），
  累積到 `STUCK_GIVE_UP_THRESHOLD` 次就暫時放棄這個位置 `STUCK_QUARANTINE_SECONDS` 秒，改找別的素材。
  這個累積失敗次數也用來決定重對準策略要換下一個——同一策略最多連續試 `REALIGN_STRATEGY_REPEAT` 次。
- 不管這一輪最後有沒有採集成功，都不等待，立刻回到「尋找素材中」找下一個目標——反正通常畫面上還有
  別的素材，原地等這一顆刷新沒有意義。

### 重對準策略（`REALIGN_STRATEGIES`）

素材的螢幕座標會隨角色移動而變（鏡頭跟著角色走），所以策略裡的每一步都是「往某個方向按住幾秒」
（`REALIGN_STEP_DURATION` 的倍數），不是換算成像素距離——這樣不管素材目前在畫面上哪個位置都適用。
每個策略是一串 `(方向角色, 時長倍率)`：

- `toward`：面向素材的方向（例如素材在下方就是「下」）
- `away`：`toward` 的反方向
- `side1` / `side2`：垂直於 `toward` 的兩個方向

預設的 5 個策略：往前一小步 → 往前+往一側（近似斜向）→ 前進後退甩開卡位 → 左右／上下側移 →
先退後再往另一側。同一個策略連續試 `REALIGN_STRATEGY_REPEAT` 次沒用才換下一個，可依實際卡住的
狀況調整每個策略的方向組合或時長。

## 運作方式（固定路線模式）

`engine.py` 的 `FixedRouteEngine` 完全不做圖像辨識，只是照著 `config.FIXED_ROUTE`
（一個步驟清單）機械化地執行：

```
從清單第一步開始 → 依序執行每一步（移動或採集）→ 清單跑完一輪 → 回到起點 → 從頭再跑一輪 → ...
```

- 每個步驟有三種：
  - `{"type": "move", "direction": ..., "duration": ...}`：往該方向按住按鈕幾秒——跟影像辨識模式
    移動時用的是同一個按壓函式，`duration` 一樣是「按住幾秒」，不是像素距離，所以角色移動速度變了
    只需要調整這個數字，不用改邏輯。
  - `{"type": "collect", "action": ..., "times": ..., "interval": ...}`：直接點擊 `COLLECT_POINTS`
    裡對應的按鈕幾次，中間不驗證有沒有真的採到東西（因為這個模式的重點就是不依賴辨識）。
  - `{"type": "go_home"}`：立刻依照目前累積的位移量走回起點（有設定 `images/home_landmark.json`
    就用地標導航，沒有就用估計位置），回到後把位移估計歸零。`FixedRouteEngine` 還支援這個步驟，
    但**目前預設的 `FIXED_ROUTE` 完全沒用到它**——見下面「回原點：目前改用固定反向路徑，不靠影像
    辨識」，地標導航實測太不穩定，暫時整個不用。
- 跑完整份清單一輪之後，一樣會回到起點，然後重新從清單第一步開始，一直循環到你按停止（現在「回到
  起點」是每一塊自己內建的固定反向路徑，見下方，不是額外呼叫 `go_home()`）。
- 這個模式沒有重對準/卡住重試機制——如果角色被地形卡住，需要自己調整 `FIXED_ROUTE` 的步驟或
  `STUCK_*` 系列參數（那些是影像辨識模式移動時共用的卡住偵測，固定路線模式的移動同樣會經過
  `_hold_point`，但不會經過 `_move_toward` 的卡住偵測迴圈，純粹照表操課）。

### 範例路線：菱形用 X 型切成四塊，各自從中心出發

`config.py` 裡的 `FIXED_ROUTE` 預設不是手寫的固定清單，而是用 `_build_diamond_piece_route()`
產生出來的——對應「採集區域是個菱形、用 X 型（菱形自己的兩條對角線）從正中心切成四塊，上、右是
樹木、左、下是礦石，移動量不同」這種情境：

- 往某個對角方向連續走（例如「左+上」重複），走出來的路徑會剛好平行菱形的一條邊；這兩組對角線同時
  也正好是菱形自己的對角線（連接對邊中點、通過正中心）。同時沿這兩條對角線切下去就是一個 X，會把
  菱形切成 4 個**指向原本 4 個角（上/右/下/左）的小菱形**——不是切成上下左右那種普通的四象限。
- **上、右**兩塊是樹木；**左、下**兩塊是礦石。
- 用對角線座標軸來看，每一塊本身就是一個方形（中心正好是這個方形的其中一角），所以掃描很單純：
  從中心開始，標準的來回掃描（boustrophedon）直接掃過去就好，不需要像切成三角形那樣「範圍逐漸
  變寬」的處理。
- 掃完一塊後，插入一個 `{"type": "go_home"}` 步驟，立刻依偏移量走回中心，再開始掃下一塊。
  順序：**上 → 右 → 左 → 下**，四塊都掃完才會（照一般規則）算完整跑完一輪、正常回到起點。
- 可調整的參數：
  - `FIXED_ROUTE_TREE_RADIUS` / `FIXED_ROUTE_TREE_STEP` / `FIXED_ROUTE_TREE_ACTION`：上、右
    （樹木）兩塊的範圍大小、每半步按住時長、採集鈕名稱（預設 `"lumber"`）。`RADIUS` 產生的網格是
    `(RADIUS+1) x (RADIUS+1)`，實測樹木是 3x3 排列，所以預設是 `RADIUS=2`（2+1=3）——如果你的樹木
    區域大小不同，照這個公式反推要設多少。
  - `FIXED_ROUTE_TREE_STEP_UP`：**「上」方向單獨的按住時長**，跟其他三個方向（用
    `FIXED_ROUTE_TREE_STEP`）分開設定。實測發現前幾格都對，但掃到離中心最遠的那一排時位置整體
    偏低，原因是遊戲裡「上」方向每按住一秒實際移動的距離似乎比其他三個方向少——如果統一加長全部
    方向的時長，下/左/右會變成走過頭，所以只單獨拉長「上」。這個機制（`_step_durations()`）目前
    只套用在樹木區；礦石區（`FIXED_ROUTE_ORE_STEP_DURATIONS`）四個方向還是統一用
    `FIXED_ROUTE_ORE_STEP`，等你實測礦石區也有類似狀況再比照辦理。
  - `FIXED_ROUTE_ORE_RADIUS` / `FIXED_ROUTE_ORE_STEP` / `FIXED_ROUTE_ORE_ACTION`：左、下
    （礦石）兩塊用，`STEP` 預設比樹木區小，因為礦石通常比較密集；`RADIUS` 目前還是預設值，等你
    實測礦石區實際排列後再調整（公式同上）。
  - **這些時長一定要先實測**：太大會一步就噴飛出去，建議先把 `FIXED_ROUTE_ORE_RADIUS` 暫時改成
    `1`（每塊只有 4 個點，跑一輪很快），觀察一步實際移動的距離，調到滿意了再改回實際要的大小。
- 移動永遠是「一組對角線（兩個半步）→ 採集」成對出現，包含一開始從中心往外走的路段也一樣——不會
  有走了一段路卻沒停下來採集的情況；而且這次每一塊內部完全不重複，不會有格子被點兩次採集鈕。
- **每一塊各自的起始偏移**：`FIXED_ROUTE_TOP_OFFSET` / `_RIGHT_OFFSET` / `_LEFT_OFFSET` /
  `_BOTTOM_OFFSET`，每個都是一串 `(方向, 秒數)`，在該塊**第一次採集之前**先執行、不算格子、不會
  採集，四塊各自獨立設定、互不影響。這裡處理兩件事：
  1. **補間隙（每一塊都需要，不是只有樹木）**：正中心本來就碰不到任何素材，最近的素材一定在某個
     距離之外，所以四塊都要先往各自的方向（指向那一塊對應的菱形角：上/右/左/下）跨出一大步，才
     開始細緻的方格掃描——不要改 `FIXED_ROUTE_TREE_STEP`／`ORE_STEP` 來處理這件事，那樣會連格子
     跟格子間的間距一起放大。四塊預設都已經加上這個跨步，方向對應各自的角，但實際距離目前是用猜的
     （只有 `TOP` 因為你已經實測過，數值比較準），跑過才知道夠不夠，不夠就把對應的秒數調大。
  2. **偏移校正（疊加在補間隙後面）**：實測發現整塊的執行結果會偏離預期方向（產生的步數本身是
     對稱的，所以這是實際遊戲裡移動距離不完全對稱造成的真實偏移，不是掃描邏輯的 bug）。**這個偏移
     每一塊不一定一樣**，所以每塊各自設定——目前只有 `RIGHT_OFFSET` 在補完間隙後多加了一點往上的
     校正，抵銷你回報的「太偏右下」；`LEFT_OFFSET`／`BOTTOM_OFFSET` 還沒有加校正，等你實測那兩塊
     後再自行補上。
  這些偏移都是真的按鍵移動，走完之後會自動算進該塊的固定反向路徑裡一起走回去（見下面「回原點」）。
- 如果你的採集區域不是菱形、不是這樣切、或想要完全自訂的路線，直接把 `FIXED_ROUTE = (...)` 那段換成
  手寫的步驟清單即可（格式跟上面「固定路線模式」描述的一樣），不影響其他部分。

### 用 `record_route.py` 直接錄一次真人操作，取代手動猜時長

上面所有的按住時長、間隙、偏移校正，本質上都是在「猜」角色實際移動的距離要按幾秒——與其這樣反覆
試錯，不如**直接把想要的路線真的玩一次，錄下你實際按鍵的時間**，直接轉成 `FIXED_ROUTE` 的格式。

這可行的前提是：這款遊戲雖然吃不到*模擬*鍵盤輸入（所以 bot 本身只能用滑鼠點畫面按鈕），但通常
**吃得到真的實體鍵盤**（畫面上會看到 `SPACE`、`LeftShift` 這種鍵盤標示，代表遊戲本身支援鍵鼠混合
操作）。`record_route.py` 就是利用這點：用 `keyboard` 套件的全域鍵盤監聽，記錄你按下/放開移動鍵的
確切時間，以及按採集鍵的次數與間隔，轉換成跟手寫 `FIXED_ROUTE` 一模一樣的格式直接印出來。

**一次錄一塊**（上/右/左/下），跟現有 `FIXED_ROUTE` 分四塊的架構對應，每一塊都是從正中心開始錄。
錄完之後**不會**加 `{"type": "go_home"}`——回原點的方式已經改成不靠影像辨識（見下面兩段「為什麼」），
直接把這塊錄到的移動**自動算出反向路徑**接在後面，讓角色照原路走回去。

用法：

1. 打開 `config.py`，把 `RECORD_MOVE_KEYS`／`RECORD_COLLECT_KEYS` 改成你實際遊戲裡用的按鍵
   （預設是 WASD + 數字鍵 `2`/`3`，如果你用方向鍵或其他按鍵要自己改對應）。
2. 執行 `run_record_route.bat`（或 `python record_route.py`），需要系統管理員權限的原因跟 `bot.py`
   一樣（`keyboard` 套件要能穩定攔截全域按鍵）。
3. 終端機會先問你**現在要錄哪一塊**（上/右/左/下，或輸入 `q` 結束並輸出結果）。選好之後站回正中心，
   按 `F1` 開始錄製，照那一塊實際想要的路線操作一次（移動、採集都用真的鍵盤按），操作完按 `F2`
   結束——結束時會自動：①把移動時長乘上 `FIXED_ROUTE_RECORDED_SCALE`（見下方原因），②算出反向
   路徑接在後面走回原點。
4. 可以重複第 3 步，一次session裡把想錄的幾塊都錄完；每塊只需要錄一次，不用四塊都錄才能結束。
5. 全部錄完後輸入 `q`，終端機會印出**每一塊各自的清單**（例如 `FIXED_ROUTE_TOP_RECORDED = [...]`）
   加上一行組合建議，同一份內容也會存到 `recorded_route.txt`。
6. 接著會問**要不要直接套用到 `config.py`**（`y`/`n`）：選 `y` 的話，工具會自動找到 `config.py`
   裡對應的 `FIXED_ROUTE_TOP`／`_RIGHT`／`_LEFT`／`_BOTTOM` 定義（不管原本是 `_build_diamond_piece_route(...)`
   算出來的、還是手寫的、還是先前錄過的，都會被整段取代），直接寫入新錄的內容——不用自己複製貼上、
   不用管行數對不對。套用後 `HOTKEY_TEST_TOP` 等單獨測試跟完整的 `FIXED_ROUTE` 都會立刻用到新版本，
   下次啟動 bot.py 就生效。選 `n` 的話就只印出來、存到 `recorded_route.txt`，自己手動貼。
7. 如果這次只想先錄一兩塊，之後想補其他塊：重新執行工具、只選還沒錄的那幾塊，套用時也只會動到
   有錄到的那幾塊，其他塊的定義不受影響——可以四塊分開、其中幾塊用錄的、幾塊用算的混著用。

細節：

- 移動鍵的「按住秒數」直接用按下到放開的實際時間，再乘上 `FIXED_ROUTE_RECORDED_SCALE`（見下方），
  不用再憑感覺猜。
- 連續按同一個採集鍵、間隔在 `RECORD_COLLECT_GROUP_GAP`（預設 1 秒）以內的，會合併成一筆
  `{"times": N, "interval": 平均間隔}`，對應「站在原地連續點好幾下採集鈕」這個動作；間隔超過就會
  被當成分開的兩個動作。

**為什麼錄到的時長重播時會超過距離、`FIXED_ROUTE_RECORDED_SCALE` 是什麼**：錄製量到的是真的鍵盤
按下/放開時間，最早 bot 重播移動是用滑鼠模擬點住畫面上的 D-pad——這是兩種不同的輸入路徑，遊戲的
實際反應速度不一樣，同樣的秒數，滑鼠模擬按住走的距離比真的按鍵盤走的距離多，所以當時把
`FIXED_ROUTE_RECORDED_SCALE` 一路從 `0.8` 調到 `0.55` 想要補償。過程中還發現一個真正的 bug：
`pyautogui`／`pydirectinput` 預設每呼叫一次函式（`moveTo`/`mouseDown`/`mouseUp`/`keyDown`/`keyUp`）
就會自動多睡 `0.1` 秒（`pyautogui.PAUSE`，給人類手動寫腳本用的節奏，這個專案完全不需要），而
`_hold_point()` 每次移動要呼叫好幾次這些函式，等於每次按住方向都平白多了 `0.1~0.2` 秒的真實按住時間，
對錄到的短動作（很多只有 `0.05~0.3` 秒）影響特別大——已在 `engine.py` 開頭把 `pyautogui.PAUSE`／
`pydirectinput.PAUSE` 都設成 `0` 修掉。另外也把 `_hold_point()` 原本拖曳到方向鈕「花 `0.05` 秒動畫
慢慢拖過去」的動作改成瞬間跳過去，測試證實遊戲不用看到拖曳動畫也認得到方向。

現在移動預設改用 `config.USE_KEYBOARD_MOVEMENT`（見上面「控制方式」）——如果掃描碼鍵盤模擬能正常
運作，就是鍵盤錄、鍵盤重播，物理上一致，理論上不太需要縮放；所以 `FIXED_ROUTE_RECORDED_SCALE` 已
重設回 `1.0`，等於先前累積的所有調整都因為「輸入方式整個換掉了」而作廢，需要在新的鍵盤重播機制下
重新測、重新調這個係數（如果偏移方向還是很一致，直接調這個數字就好，不用重錄）。

`_reverse_home_moves()` 反推回原點的方法就是單純「方向對調、時長照抄」——上下左右的移動距離都跟按
壓秒數成正比，四個方向速度一樣，所以去程跟回程只要方向相反、秒數相同，就會精準抵銷回到原點，不需要
額外的速度校正。

### 「中心點」是什麼、人物移動後怎麼回去（影像辨識模式）

> **固定路線模式現在不用這一套**——`go_home()`（不管是地標導航還是估計位置）目前只有影像辨識模式
> 還在用（找不到素材、回中心待機時）；固定路線模式已經改成每塊各自內建的固定反向路徑，見上面
> 「用 record_route.py 直接錄一次真人操作」跟下一節，完全不涉及影像辨識，也不會有這裡講的估計誤差
> 問題。這一節保留給還在用 `go_home()` 的影像辨識模式參考。

這裡的「中心點」不是螢幕上某個固定像素位置，而是**角色在遊戲世界裡的某個位置**。
`screen_center()` 回傳的是螢幕（或 `SCAN_REGION`）正中央的像素座標，這是固定不變的，用來代表「角色目前所在
的螢幕位置」——因為這款遊戲的鏡頭會跟著角色走，角色永遠畫在畫面中間附近，所以可以拿這個固定像素點當作
「角色現在在哪」的替代品，並不是真的在讀角色的世界座標。角色離開這個位置之後，`go_home()` 分兩段回去：

1. **固定模式（估計位置，一定會跑）**：把「你按下 F1 啟動當下角色站的位置」當中心，用「往某個方向按住
   按鈕的秒數」累加成一個估計位移（`engine._position`，出發時歸零）：每次移動/重對準/脫困，都會依按住
   的方向與時間更新這個估計值，回中心待機時就往相反方向按住同樣的秒數，把估計值歸零。這只是「用時間換算
   距離」的估算法，不是真的座標系統——同樣按住 1 秒，每次角色實際移動的距離不一定完全一樣（遊戲本身處理
   輸入的時間點就不是絕對穩定的，加上卡地形、被打斷移動等因素），所以單獨用這個方法回中心，位置會有落差、
   長時間跑下來還可能累積誤差。
2. **地標校正（`images/home_landmark.json` 存在時，接在第 1 步後面跑）**：在想當中心點的位置，用
   `capture_helper.py` 的 `home_landmark` 類別截一個固定不動、獨特的地標（石頭、建築物角落、特殊地面
   花紋都可以），工具會自動算出這個地標的中心跟畫面正中央的偏移量，寫進 `images/home_landmark.json`
   ——不用手動填任何 config 數值。第 1 步的估計走完之後，如果這個檔案存在，就會去找這個地標目前在畫面
   上的位置，只要它跟「當初截圖時的偏移量」對不上，就照著找素材同一套邏輯（`_move_toward`）移動過去，
   直到對上為止——這是用畫面上看得到的東西做最後校正，把第 1 步「用時間換算距離」累積的誤差修正掉，
   當作精度的保底。如果角色離地標太遠、地標整個不在畫面上，這一步就會直接跳過，維持第 1 步估計出來的
   位置（等下一次待機掃描，通常已經被第 1 步帶得夠近，地標就找得到了）。

  影像辨識模式的「上」方向實測也比其他三個方向慢，同樣的估計誤差理論上也會發生在這裡——如果觀察到
  影像辨識模式回中心待機時衝過頭，一樣建議照上面設一次地標。（固定路線模式已經不受這個問題影響，見
  上面的提示框。）

## 素材截圖與辨識

- `capture_helper.py`：截圖工具。啟動後（或按 `M`）會先問你現在要截哪個素材／面向
  （清單在檔案開頭的 `CATEGORIES`，跟 `config.py` 的素材/面向名稱對應，也包含 `home_landmark`
  中心點地標），之後存檔會自動用 `<名稱>_<編號>.png` 命名，編號會接續 `images/` 資料夾裡已經有的
  檔案，不會覆蓋、也不用你自己改檔名。操作：左鍵加點框出多邊形、右鍵/Enter 存檔（存出 `.png` +
  `_mask.png` 遮罩）、`G` 重新截一張畫面（拍下一個動畫幀/時間點時用）、`M` 換素材、`Q` 離開。
  截 `home_landmark` 時會額外自動算好跟畫面中心的偏移量、寫進 `images/home_landmark.json`，不用手動填。
- `test_recognition.py`：離線測試工具，不用真的跑機器人。
  `python test_recognition.py <截圖> <模板> [門檻]`，視窗內用 `+`/`-` 調門檻、`R` 重跑，
  可以看到比對分數跟熱圖，用來決定 `config.MATCH_THRESHOLD` 設多少比較穩。
- 同一顆素材（或同一個面向）可以放**多張模板**：`config.py` 的 `image` 欄位除了單一路徑字串，
  也可以放路徑清單，例如素材有搖晃動畫或不同光影下外觀不同：
  ```python
  "image": ["images/tree_big_1.png", "images/tree_big_2.png"],
  ```
  比對時任一張命中就算數，也正好對應 `capture_helper.py` 自動編號存出來的檔名。

## config.py 設定

| 設定 | 說明 |
|---|---|
| `HOTKEY_START` / `HOTKEY_STOP` / `HOTKEY_DEBUG` | 啟動／停止／讀取滑鼠座標熱鍵（讀你自己的實體鍵盤，跟遊戲吃不吃鍵盤無關） |
| `HOTKEY_TEST_TOP` / `_RIGHT` / `_LEFT` / `_BOTTOM` | 固定路線模式專用：單獨測試上/右/左/下其中一塊（預設 F5~F8），停止一樣用 `HOTKEY_STOP` |
| `DEBUG_POSITION_DELAY` | 按下 `HOTKEY_DEBUG` 後，等幾秒才讀取滑鼠座標（留時間把滑鼠移過去） |
| `MATERIALS` | 素材清單，每項要有 `image`（路徑或清單）、`collect_action`（對應 `COLLECT_POINTS` 的名稱）、`collect_times`（點幾次）、`collect_interval`（每次間隔秒數） |
| `MATCH_THRESHOLD` | 圖像比對信心門檻（0~1），用 `test_recognition.py` 調 |
| `SCAN_REGION` | 只掃描螢幕的一部分，`None` 代表全螢幕 |
| `USE_KEYBOARD_MOVEMENT` | 移動要用鍵盤（`pydirectinput` 掃描碼）還是滑鼠拖方向鈕（預設 `True`＝鍵盤），見上面「控制方式」 |
| `MOVE_KEYS` | `USE_KEYBOARD_MOVEMENT=True` 時，上下左右對應的鍵盤按鍵（預設 WASD） |
| `MOVE_POINTS` | `USE_KEYBOARD_MOVEMENT=False` 時，上下左右方向按鈕的螢幕座標（滑鼠按住＝持續朝該方向移動＋轉向） |
| `COLLECT_POINTS` | 具名採集按鈕的螢幕座標，`MATERIALS` 用 `collect_action` 指定要點哪一個 |
| `JUMP_POINT` | 選用的跳躍按鈕座標，脫困時用；沒有就留 `None` |
| `SCAN_WHILE_MOVING` | 邊走邊重新掃描素材位置的頻率（秒） |
| `REACH_RADIUS` | 距離角色（畫面中心）幾 px 內視為已到達素材旁邊 |
| `COLLECT_VERIFY_DELAY` / `COLLECT_RETRY_MAX` | 按完採集鈕等幾秒確認消失／最多重試幾次 |
| `REALIGN_STEP_DURATION` | 重對準策略每一步的基準按住時長（秒），策略裡用倍率調整 |
| `REALIGN_STRATEGIES` / `REALIGN_STRATEGY_REPEAT` | 重對準策略清單，以及同一策略連續嘗試幾次才換下一個（見上面「重對準策略」） |
| `CHARACTER_CROP` / `FACING_IMAGES` | 進階、選用：設定角色在螢幕上的範圍 + 朝上下左右的定格截圖，開啟後同一顆素材連續失敗多次時會改用截圖判定面向（沒設定就一直用按鈕估計） |
| `SAME_MATERIAL_TOLERANCE` | 幾 px 內視為「同一顆」素材（用來累計持續失敗次數） |
| `STUCK_FAIL_THRESHOLD` | 同一顆累積失敗幾次後，優先改用截圖判定面向 |
| `STUCK_GIVE_UP_THRESHOLD` / `STUCK_QUARANTINE_SECONDS` | 累積失敗幾次後暫時放棄、放棄後幾秒內不再選它 |
| `STUCK_TIMEOUT` / `STUCK_MOVEMENT_THRESHOLD` / `STUCK_ESCAPE_DURATION` / `STUCK_MAX_ATTEMPTS` | 移動中卡住的偵測與脫困參數 |
| `SCAN_INTERVAL` | 找不到素材、用遊走模式時，每輪掃描間隔 |
| `RETURN_HOME_WHEN_IDLE` / `IDLE_SCAN_INTERVAL` | 找不到素材時是否回中心待機、待機時多久掃描一次 |
| `HOME_REACH_RADIUS` | 地標校正時，離目標幾 px 內算「已對準」（見上面「中心點是什麼」）——是否跑地標校正由 `images/home_landmark.json` 存不存在決定，不用另外設定 |
| `SEARCH_DURATION` / `SEARCH_PATTERN` | `RETURN_HOME_WHEN_IDLE = False` 時，遊走找素材的時長與路線 |
| `APPROACH_MISS_TOLERANCE` | 移動靠近時，容忍連續幾次沒偵測到素材才判定「素材消失」放棄（避免單一 tick 的誤判就中斷） |
| `FIXED_ROUTE` | 固定路線模式跑的完整清單（四塊 + 中間的 `go_home`），`{"type": "move"/"collect"/"go_home", ...}` 步驟格式，預設由 `_build_diamond_piece_route()` 產生，見上面「範例路線：菱形用 X 型切成四塊」 |
| `FIXED_ROUTE_TOP` / `_RIGHT` / `_LEFT` / `_BOTTOM` | 上/右/左/下各自單獨的步驟清單，每個都已經內建「去程 + 自動算出的反向回程」，不需要額外的 `go_home`（`FIXED_ROUTE` 就是這四個接起來的），給 `HOTKEY_TEST_*` 單獨測試用。目前 `_TOP` 是演算法產生的、`_RIGHT`／`_LEFT`／`_BOTTOM` 已經換成 `record_route.py` 錄出來的真人操作資料——可以用 `record_route.py` 的一鍵套用功能覆蓋，或手動貼上取代 |
| `FIXED_ROUTE_TREE_RADIUS` / `_STEP` / `_ACTION` | 菱形範例路線上、右（樹木）兩塊的範圍大小、每半步按住時長、採集鈕名稱 |
| `FIXED_ROUTE_ORE_RADIUS` / `_STEP` / `_ACTION` | 菱形範例路線左、下（礦石）兩塊用，同上三項 |
| `FIXED_ROUTE_COLLECT_TIMES` / `_INTERVAL` | 菱形範例路線每格的點擊次數、點擊間隔（四塊共用） |
| `FIXED_ROUTE_TOP_OFFSET` / `_RIGHT_OFFSET` / `_LEFT_OFFSET` / `_BOTTOM_OFFSET` | 每一塊各自開始掃描前的補間隙／偏移校正，見上面「範例路線」說明 |
| `APPROACH_DISAPPEAR_ARRIVAL_FACTOR` | 目標消失時，若消失前最後量到的距離在 `REACH_RADIUS` 的幾倍以內就視為已抵達（而不是判定失敗），適用素材靠近與地標回中心兩種情境 |
| `RECORD_MOVE_KEYS` / `RECORD_COLLECT_KEYS` / `RECORD_COLLECT_GROUP_GAP` | `record_route.py` 專用：實體鍵盤按鍵對應的方向／採集動作名稱、連續採集按鍵合併成一筆的間隔上限，見上面「用 record_route.py 直接錄一次真人操作」 |
| `FIXED_ROUTE_RECORDED_SCALE` | 錄到的移動時長要乘上的縮放係數（目前 `0.9`，改用鍵盤重播後從 `1.0` 微調），補償重播跟錄製距離不一致的問題，見上面「為什麼錄到的時長重播時會超過距離」 |
| `FIXED_ROUTE_REST_SECONDS` | 單獨測試一塊（`F5`/`F6`/`F7`/`F8`）跑完回到起點後，休息幾秒再跑下一輪（目前 `780`＝13 分鐘） |
| `FIXED_ROUTE_FULL_REST_SECONDS` | 完整路線（`F1`，上右左下四塊都跑完）回到起點後，休息幾秒再開始下一輪（目前 `780`＝13 分鐘） |

## Log 與除錯

- GUI 視窗裡的黑色文字框：即時顯示目前在做什麼。
- `bot_log.txt`（跟 `bot.py`/打包後的 exe 同一層目錄）：所有訊息都會加上時間戳記存一份，
  關掉程式也不會消失，方便測完之後回頭看發生了什麼。

常見訊息代表的意思：

| Log 訊息 | 代表 | 可以調的設定 |
|---|---|---|
| 一直「搜尋中：遊走一圈」找不到素材 | 圖像辨識沒命中 | 用 `test_recognition.py` 檢查模板/門檻，或截圖檔名路徑對不上 |
| 反覆「重對準：轉向面對素材」 | 面向一直判斷錯（估計可能跟遊戲實際不同步） | 設定 `CHARACTER_CROP`/`FACING_IMAGES` 改用截圖判定 |
| 「這個位置已連續失敗 N 次，重對準改用截圖判定面向」 | 觸發 `STUCK_FAIL_THRESHOLD` | 確認有沒有設定截圖面向；沒設定的話這行只是提示，實際還是用估計 |
| 「這個位置累積失敗 N 次，暫時放棄 N 秒」 | 觸發 `STUCK_GIVE_UP_THRESHOLD` | 太常出現代表某個點位有問題，或調整 `STUCK_GIVE_UP_THRESHOLD`/`STUCK_QUARANTINE_SECONDS` |
| 「卡住，第 N 次脫困」很多次 | 移動路徑被地形卡住 | 調 `STUCK_TIMEOUT`/`STUCK_ESCAPE_DURATION` |

## 打包成 exe（選用）

Windows 上可以用 `build.bat`（呼叫 PyInstaller）打包成獨立資料夾，
或用 `installer.iss`（Inno Setup）做成安裝檔。兩者都會把 `config.py` 和 `images/` 放在
輸出資料夾旁邊，方便使用者之後直接編輯，不用重新打包。

## 已知限制

- 回中心的「固定模式」（按住按鈕秒數估算）不是真的座標，同樣的按住時長每次實際移動的距離不一定一樣，
  長時間跑下來可能會累積誤差（例如卡地形、被打斷移動、遊戲處理輸入的時間點本身就不穩定）。詳見上面
  「中心點是什麼」；沒有 `images/home_landmark.json` 時，回中心完全只靠這個估算，建議截一張地標圖讓
  地標校正接手做最後修正。
- 角色面向預設用按鈕估計；`CHARACTER_CROP`/`FACING_IMAGES` 是可選的加強，沒截圖設定就不會啟用。
- 滑鼠只有一個游標，同一時間只能按住一個方向按鈕，沒辦法像鍵盤那樣同時按兩個方向做真正的斜向移動，
  只能用「先修正距離較遠的那一軸」來逼近。
- `MOVE_POINTS`/`COLLECT_POINTS`/`JUMP_POINT` 的座標、以及地標圖裡記錄的偏移量，都綁定目前的螢幕
  解析度和遊戲視窗位置/大小，換了就要用 `F3`（Debug 座標）重新抓按鈕座標；地標圖也要重新截一次。
